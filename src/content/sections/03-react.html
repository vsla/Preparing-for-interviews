3. React (Deep) 6 topics

React is the dominant UI library in frontend interviews. Beyond JSX syntax, interviewers focus on reconciliation, fiber architecture, hooks lifecycle, performance optimization, and state management tradeoffs. Knowing why React works the way it does is as important as knowing how to use it.

3.1 Core Concepts

JSX Compilation

// JSX is syntactic sugar for React.createElement (classic runtime)
// or just _jsx (new automatic runtime — React 17+)

// Before (classic):
const el = <div className="box">Hello</div>;
// Compiles to:
const el = React.createElement("div", { className: "box" }, "Hello");

// React 17+ automatic runtime (no need to import React):
import { jsx as _jsx } from "react/jsx-runtime";
const el = _jsx("div", { className: "box", children: "Hello" });

Virtual DOM & Reconciliation

React maintains a virtual DOM (lightweight JS representation). When state changes, React creates a new vDOM tree and diffs it against the previous one (reconciliation). Only the minimal set of real DOM changes are applied (commit phase).

Fiber Architecture

React Fiber (React 16+) is a rewrite of the reconciliation engine. Each component has a "fiber" — a JS object with component info, pending updates, and links to parent/child/sibling fibers. Fiber enables:

  • Incremental rendering — split work into chunks, pause and resume
  • Prioritization — urgent updates (typing) beat non-urgent (data fetch)
  • Concurrent features — startTransition, Suspense

React 18: Concurrent Features

// Automatic batching: state updates in promises, timeouts now batch
// React 17: only updates in event handlers were batched
setTimeout(() => {
  setCount(c => c + 1);
  setFlag(f => !f);
  // React 18: ONE re-render. React 17: TWO re-renders
}, 1000);

// startTransition: mark update as non-urgent
import { startTransition } from "react";
startTransition(() => {
  setSearchResults(filter(data, query)); // can be interrupted
});
// Urgent: setQuery (input stays responsive)
// Non-urgent: setSearchResults (can be deferred)

3.2 Component Patterns

Controlled vs Uncontrolled

// Controlled: React owns the value
function ControlledInput() {
  const [value, setValue] = useState("");
  return <input value={value} onChange={e => setValue(e.target.value)} />;
}

// Uncontrolled: DOM owns the value, ref to read it
function UncontrolledInput() {
  const ref = useRef();
  const submit = () => console.log(ref.current.value);
  return <input ref={ref} defaultValue="initial" />;
}

Compound Components

// Components share implicit state via Context
const TabContext = createContext();

function Tabs({ children, defaultTab }) {
  const [active, setActive] = useState(defaultTab);
  return (
    <TabContext.Provider value={{ active, setActive }}>
      <div>{children}</div>
    </TabContext.Provider>
  );
}
Tabs.Tab = function Tab({ id, children }) {
  const { active, setActive } = useContext(TabContext);
  return <button onClick={() => setActive(id)} aria-selected={active === id}>{children}</button>;
};
Tabs.Panel = function Panel({ id, children }) {
  const { active } = useContext(TabContext);
  return active === id ? <div>{children}</div> : null;
};

// Usage:
<Tabs defaultTab="a">
  <Tabs.Tab id="a">Tab A</Tabs.Tab>
  <Tabs.Tab id="b">Tab B</Tabs.Tab>
  <Tabs.Panel id="a">Content A</Tabs.Panel>
  <Tabs.Panel id="b">Content B</Tabs.Panel>
</Tabs>

Portals & Error Boundaries

// Portals: render children outside parent DOM
import { createPortal } from "react-dom";
function Modal({ children }) {
  return createPortal(children, document.getElementById("portal-root"));
}

// Error Boundaries: catch render errors (class components only)
class ErrorBoundary extends React.Component {
  state = { hasError: false, error: null };
  static getDerivedStateFromError(error) { return { hasError: true, error }; }
  componentDidCatch(error, info) { logError(error, info); }
  render() {
    if (this.state.hasError) return <h2>Something went wrong.</h2>;
    return this.props.children;
  }
}
3.3 Hooks — Every Hook Explained

useState

const [state, setState] = useState(initialValue);
// initialValue can be a function (lazy init — runs once)
const [data, setData] = useState(() => JSON.parse(localStorage.getItem("data")));

// Functional update (safe when next state depends on prev)
setCount(prev => prev + 1);

// Objects: must spread — React doesn't merge like class setState
setUser(prev => ({ ...prev, name: "Alice" }));

// Common mistake: stale closure
const [count, setCount] = useState(0);
const handleClick = () => {
  setTimeout(() => {
    setCount(count + 1); // stale! count is captured at render
  }, 1000);
};
// Fix: functional update
setTimeout(() => setCount(c => c + 1), 1000);

useEffect

// Runs after render, after paint (asynchronous)
useEffect(() => {
  // side effect
  const sub = subscribe(userId);
  return () => sub.unsubscribe(); // cleanup: runs before next effect or unmount
}, [userId]); // dependency array

// No deps array: runs after every render
useEffect(() => { document.title = `Count: ${count}`; });

// Empty deps: runs once on mount
useEffect(() => { fetchInitialData(); }, []);

// Strict Mode runs effects TWICE in development to detect side effects

// Common mistake: missing dependency
useEffect(() => {
  const id = setInterval(() => setCount(count + 1), 1000);
  return () => clearInterval(id);
}, []); // BUG: count is stale (always 0 in closure)
// Fix: setCount(c => c + 1) or add count to deps

useLayoutEffect

// Fires synchronously after DOM mutations but BEFORE paint
// Use for: reading DOM measurements, preventing flash of wrong content

useLayoutEffect(() => {
  const { height } = ref.current.getBoundingClientRect();
  setHeight(height); // safe — user won't see flicker
}, []);

// useEffect fires AFTER paint — user might see intermediate state

useRef

// 1. DOM reference
const inputRef = useRef(null);
<input ref={inputRef} />
inputRef.current.focus();

// 2. Mutable container (doesn't trigger re-render)
const renderCount = useRef(0);
renderCount.current++; // mutation, not state

// 3. Previous value
const prevValue = useRef(value);
useEffect(() => { prevValue.current = value; }, [value]);
// prevValue.current holds previous render's value

// 4. Stable callback reference (avoid recreating intervals etc.)
const callbackRef = useRef(callback);
useLayoutEffect(() => { callbackRef.current = callback; }, [callback]);

useMemo

// Memoize expensive computation — recomputes when deps change
const sortedList = useMemo(
  () => [...list].sort((a, b) => a.name.localeCompare(b.name)),
  [list]
);

// Also used to stabilize object references for deps
const config = useMemo(() => ({ url, timeout }), [url, timeout]);

// DON'T use useMemo for cheap computations
// DON'T use it "just in case" — adds overhead, hides perf issues

useCallback

// Memoize function reference — same function unless deps change
const handleSubmit = useCallback(async (data) => {
  await submitForm(data);
  navigate("/success");
}, [navigate]);

// Useful when:
// 1. Passing to memoized child: <Button onClick={handleSubmit} />
// 2. Using function as useEffect dependency

// useCallback(fn, deps) === useMemo(() => fn, deps)

useReducer

// Better than useState for complex state transitions
const initialState = { count: 0, step: 1, history: [] };

function reducer(state, action) {
  switch (action.type) {
    case "INCREMENT":
      return { ...state, count: state.count + state.step, history: [...state.history, state.count] };
    case "SET_STEP":
      return { ...state, step: action.payload };
    case "RESET":
      return initialState;
    default:
      throw new Error(`Unknown action: ${action.type}`);
  }
}

const [state, dispatch] = useReducer(reducer, initialState);
dispatch({ type: "INCREMENT" });
dispatch({ type: "SET_STEP", payload: 5 });

useSyncExternalStore

// For subscribing to external stores safely (SSR-safe)
import { useSyncExternalStore } from "react";

function useOnlineStatus() {
  return useSyncExternalStore(
    (callback) => {  // subscribe
      window.addEventListener("online", callback);
      window.addEventListener("offline", callback);
      return () => {
        window.removeEventListener("online", callback);
        window.removeEventListener("offline", callback);
      };
    },
    () => navigator.onLine,  // getSnapshot (client)
    () => true               // getServerSnapshot (SSR)
  );
}

Custom Hooks

// Must start with "use" — enables linting rules
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const controller = new AbortController();
    setLoading(true);
    fetch(url, { signal: controller.signal })
      .then(r => r.json())
      .then(setData)
      .catch(e => { if (e.name !== "AbortError") setError(e); })
      .finally(() => setLoading(false));
    return () => controller.abort();
  }, [url]);

  return { data, loading, error };
}

function useLocalStorage(key, initial) {
  const [value, setValue] = useState(() => {
    try { return JSON.parse(localStorage.getItem(key)) ?? initial; }
    catch { return initial; }
  });
  const set = useCallback(v => {
    setValue(v);
    localStorage.setItem(key, JSON.stringify(v));
  }, [key]);
  return [value, set];
}

  setCount(count + 1); // all use same stale count!

// Fix: functional update

  setCount(c => c + 1); // each builds on previous
// Q: Key prop in lists — why can't we use index?
// Index as key causes bugs when list is reordered/filtered:
// React sees key=0 and reuses the DOM node/component state
// Fix: use stable, unique IDs from data

// Q: When does useEffect cleanup run?
// 1. Before the effect runs again (deps changed)
// 2. When component unmounts
// Order: render → cleanup (if needed) → effect

// Q: Why is my state update not reflected immediately after setState?

  setState(42);

};

⚡ Key Takeaways

  • Reconciliation uses the fiber tree; elements with the same type at the same position are updated in place
  • Never mutate state directly — always use the setter or produce a new object/array
  • useEffect with an empty dependency array runs once after mount, equivalent to componentDidMount
  • Keys in lists must be stable, unique identifiers — array indexes cause bugs when the list reorders
  • useMemo and useCallback are for referential stability and expensive work, not default practice
  • Context re-renders all consumers on every value change; split contexts or use selectors to limit scope
  • useReducer is preferable when multiple state values are interdependent
  • The rules of hooks (no conditions, no loops) exist because hooks depend on call order across renders
