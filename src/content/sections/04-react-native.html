4. React Native (Deep) 7 topics

React Native interviews combine React knowledge with mobile platform specifics. Key topics are the JavaScript bridge, the new Fabric/JSI architecture, navigation patterns, performance on low-end devices, and platform-specific styling. Expo vs bare workflow tradeoffs come up frequently.

4.1 Architecture

Old Architecture

JavaScript Thread → Bridge (async, JSON-serialized messages) → Native Thread. The bridge was a bottleneck: async only, serialization overhead, can't share memory.

New Architecture (React Native 0.68+)

  • JSI (JavaScript Interface): Direct C++ binding. JS can hold references to native objects and call methods synchronously without serialization.
  • Fabric: New rendering system. C++ shadow tree, synchronous layout measurement, concurrent features support.
  • TurboModules: Lazy-loaded native modules via JSI. Only loaded when first used.
  • Codegen: TypeScript/Flow specs generate native code automatically.

Hermes Engine

Bytecode-compiled JS engine optimized for React Native. Faster startup (pre-compiled), lower memory, built-in debugger. Default since RN 0.70.

Yoga Layout Engine

Cross-platform C++ Flexbox implementation used to calculate layouts. Runs on native thread. Implements a subset of CSS Flexbox.

⚡ Key Takeaways

  • Old architecture: JS thread communicates with native via serialized bridge — the main performance bottleneck
  • New architecture (Fabric + JSI): synchronous native calls via C++ host objects, no serialization overhead
  • FlatList virtualizes off-screen items; never use a plain ScrollView for long lists
  • Styles are inline JS objects resolved to native constraints — no cascade, no specificity rules
  • React Navigation stack navigators preserve state; tab navigators mount all screens on initialization
  • Platform.OS and platform-specific file extensions (.ios.js / .android.js) are the two branching strategies
  • InteractionManager.runAfterInteractions defers heavy work until after animations complete
