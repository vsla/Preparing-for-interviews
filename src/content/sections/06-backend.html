TLS Handshake (TLS 1.3)

1. Client Hello (supported ciphers, random) → 2. Server Hello (chosen cipher, certificate, random) → 3. Client verifies cert → 4. Key exchange (ECDHE) → 5. Both derive session keys → Encrypted communication. TLS 1.3: 1 RTT (0-RTT for resumption).

6.2 API Design

REST Principles

  • Uniform Interface: consistent resource-based URLs
  • Stateless: no server-side client state
  • Cacheable: responses can be cached
  • Client-Server: separation of concerns
  • Layered System: intermediaries (LB, cache, gateway)

6.4 Caching 
// Cache-Control response headers
// Cache-Control: no-store              — Never cache (sensitive data)
// Cache-Control: no-cache              — Revalidate before serving cached
// Cache-Control: private, max-age=300  — Browser only, 5 min
// Cache-Control: public, max-age=86400 — CDN + browser, 1 day
// Cache-Control: public, immutable, max-age=31536000 — versioned assets

// ETags: server sends hash of content
// Client sends: If-None-Match: "etag-value"
// Server: 304 Not Modified if unchanged

// Redis cache-aside pattern
async function getUser(id) {
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);
  const user = await db.query("SELECT * FROM users WHERE id = $1", [id]);
  await redis.setex(`user:${id}`, 3600, JSON.stringify(user));
  return user;
}

// Cache invalidation
await redis.del(`user:${id}`);  // on user update

// Write-through: write to cache AND DB simultaneously
// Write-back: write to cache, async flush to DB (risk of data loss)

⚡ Key Takeaways

  • HTTP methods have semantic meaning: GET is idempotent and safe; POST is neither; PUT/DELETE are idempotent
  • REST resource naming uses nouns, not verbs; versioning belongs in the URL (/v1/) or Accept header
  • JWTs are signed, not encrypted — never put sensitive data in the payload; always verify the signature server-side
  • Cache-Control headers control both browser and CDN caching; no-store prevents all caching
  • Message queues (Kafka, RabbitMQ) decouple producers from consumers and absorb traffic spikes
  • Microservices trade deployment independence for distributed systems complexity — choose based on team size and scale
  • Implement rate limiting at the gateway or load balancer level, not inside each service
