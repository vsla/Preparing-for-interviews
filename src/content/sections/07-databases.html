7. Databases (Deep) 4 topics

Database questions assess your ability to choose and operate storage systems correctly. SQL query writing, index internals, the ACID properties, normalization, and NoSQL tradeoffs are core topics. Redis patterns, connection pooling, and the N+1 query problem come up often in practical interviews.

Indexes

-- B-tree (default): equality + range queries
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at DESC);

-- Partial index: index subset of rows
CREATE INDEX idx_active_users ON users(email) WHERE active = true;

-- GIN: full-text search, jsonb, arrays
CREATE INDEX idx_posts_tsv ON posts USING GIN(to_tsvector('english', content));
CREATE INDEX idx_tags ON posts USING GIN(tags);

-- Covering index: include extra columns to avoid heap fetch
CREATE INDEX idx_orders ON orders(user_id) INCLUDE (total, status);

EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT * FROM orders WHERE user_id = 1;

Window Functions

SELECT
  user_id, amount,
  ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn,
  RANK()       OVER (ORDER BY amount DESC),
  LAG(amount)  OVER (PARTITION BY user_id ORDER BY created_at),
  LEAD(amount) OVER (PARTITION BY user_id ORDER BY created_at),
  SUM(amount)  OVER (PARTITION BY user_id
                     ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total
FROM orders;

-- Latest order per user
SELECT * FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY created_at DESC) AS rn
  FROM orders
) r WHERE rn = 1;

CTEs and Recursive CTEs

WITH recent AS (
  SELECT * FROM orders WHERE created_at > NOW() - INTERVAL '30 days'
),
totals AS (
  SELECT user_id, SUM(amount) AS total FROM recent GROUP BY user_id
)
SELECT u.name, t.total FROM users u JOIN totals t ON u.id = t.user_id;

-- Recursive CTE: org chart
WITH RECURSIVE tree AS (
  SELECT id, name, manager_id, 0 AS depth
  FROM employees WHERE manager_id IS NULL
  UNION ALL
  SELECT e.id, e.name, e.manager_id, t.depth + 1
  FROM employees e JOIN tree t ON e.manager_id = t.id
)
SELECT * FROM tree;

JOINs

-- INNER: matching rows only
SELECT u.name, o.total FROM users u INNER JOIN orders o ON u.id = o.user_id;
-- LEFT: all left rows + matched right (NULL if no match)
SELECT u.name, COUNT(o.id) FROM users u LEFT JOIN orders o ON u.id = o.user_id GROUP BY u.id;
-- SELF JOIN: employee to manager
SELECT e.name AS employee, m.name AS manager
FROM employees e LEFT JOIN employees m ON e.manager_id = m.id;

Normalization

1NF: atomic values, no repeating groups.
2NF: 1NF + no partial dependencies on composite PK.
3NF: 2NF + no transitive dependencies (non-key cols depend only on PK).
Denormalization: trade write complexity for read performance.

JSONB, Partitioning, Replication

-- JSONB column
SELECT data->>'name' FROM products;
SELECT * FROM products WHERE data @> '{"category": "electronics"}';
CREATE INDEX ON products USING GIN(data);

-- Range partitioning
CREATE TABLE orders (order_date DATE) PARTITION BY RANGE (order_date);
CREATE TABLE orders_2024 PARTITION OF orders FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

-- pg_stat_statements: find slow queries
SELECT query, calls, mean_exec_time FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10;
7.2 NoSQL — MongoDB 
// CRUD
db.users.insertOne({ name: "Alice", age: 30, tags: ["admin"] });
db.users.findOne({ email: "a@b.com" });
db.users.find({ age: { $gte: 18 }, active: true }).sort({ name: 1 }).limit(10);
db.users.updateOne({ _id: id }, { $set: { name: "Bob" }, $push: { tags: "user" } });
db.users.deleteMany({ active: false });

// Aggregation pipeline
db.orders.aggregate([
  { $match: { status: "completed" } },
  { $group: { _id: "$userId", total: { $sum: "$amount" }, count: { $count: {} } } },
  { $sort: { total: -1 } },
  { $limit: 10 },
  { $lookup: { from: "users", localField: "_id", foreignField: "_id", as: "user" } },
  { $unwind: "$user" },
  { $project: { userName: "$user.name", total: 1 } }
]);

// Indexes
db.users.createIndex({ email: 1 }, { unique: true });
db.posts.createIndex({ title: "text", content: "text" });

⚡ Key Takeaways

  • ACID: Atomicity (all or nothing), Consistency (valid state), Isolation (concurrent txns don't interfere), Durability (committed data survives crashes)
  • Composite indexes are only used when the leftmost column is in the query — column order matters critically
  • N+1 query problem: fetching a list then querying each item individually; solve with JOIN, include, or DataLoader
  • EXPLAIN ANALYZE reveals actual execution costs; check for sequential scans on large tables
  • Redis sorted sets (ZADD, ZRANGE, ZRANK) are the canonical solution for leaderboards
  • MongoDB: embed data that is read together, reference data that is updated independently
  • Connection pooling is essential in production; pg.Pool / Mongoose pool prevent connection exhaustion
