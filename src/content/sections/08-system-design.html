Circuit Breaker Pattern

// CLOSED: requests pass through, monitoring error rate
// OPEN: all requests fail fast after threshold exceeded
// HALF-OPEN: probe request; success closes, failure reopens

class CircuitBreaker {
  constructor(fn, { threshold = 5, timeout = 60000 } = {}) {
    this.fn = fn; this.state = "CLOSED"; this.failures = 0;
    this.threshold = threshold; this.timeout = timeout;
  }
  async call(...args) {
    if (this.state === "OPEN") throw new Error("Circuit OPEN");
    try {
      const result = await this.fn(...args);
      this.failures = 0; this.state = "CLOSED";
      return result;
    } catch (err) {
      this.failures++;
      if (this.failures >= this.threshold) {
        this.state = "OPEN";
        setTimeout(() => { this.state = "HALF-OPEN"; }, this.timeout);
      }
      throw err;
    }
  }
}

8.4 CAP Theorem

Consistency: Every read returns the most recent write or an error.
Availability: Every request receives a response (not necessarily most recent).
Partition Tolerance: System continues despite network partitions.

In distributed systems you must tolerate partitions. Real choice: CP vs AP.

SystemTypeTrade-off
PostgreSQL (single node)CANot distributed
HBase, ZooKeeper, etcdCPMay reject reads during partition
Cassandra, DynamoDB, CouchDBAPMay return stale data

PACELC: Even without partition, there is a Latency vs Consistency trade-off.

⚡ Key Takeaways

  • Always clarify requirements, estimate scale, and identify bottlenecks before proposing an architecture
  • CAP theorem: distributed systems can guarantee at most two of Consistency, Availability, Partition Tolerance
  • Horizontal scaling + stateless services + load balancer is the standard baseline for high-traffic systems
  • CDNs serve static assets from edge nodes — they reduce latency and offload the origin significantly
  • Database read replicas and read-through caches handle read-heavy workloads without scaling writes
  • Idempotency keys prevent duplicate operations when clients retry failed requests
  • Event-driven architecture decouples services but requires careful handling of ordering and at-least-once delivery
