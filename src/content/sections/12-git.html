12. Git & Workflow 1 topic

Git workflow questions assess day-to-day professional habits. Common topics include rebasing vs merging, cherry-pick, conflict resolution, and branching strategies like trunk-based development or GitFlow. Knowing how to recover from mistakes — reflog, reset, revert — is a strong signal of experience.

12.1 Core Git Concepts & Commands

Merge vs Rebase

# Merge: creates merge commit, preserves history
git checkout main
git merge feature/login  # merge commit created, diverge is visible in log

# Rebase: moves feature commits to tip of target, linear history
git checkout feature/login
git rebase main  # replays feature commits on top of current main

# Golden Rule of Rebase:
# NEVER rebase public/shared branches (rewrites history = new commit SHAs)
# Safe: rebase local feature branches before creating PR

# Interactive rebase: rewrite last N commits
git rebase -i HEAD~4
# pick   = use commit as-is
# reword = change commit message
# edit   = pause to amend commit (git commit --amend, then git rebase --continue)
# squash = combine with previous commit (keep both messages)
# fixup  = combine, discard this commit's message
# drop   = remove commit entirely

Useful Commands

# Stash: temporarily save uncommitted work
git stash push -m "WIP: login feature"
git stash list                   # list all stashes
git stash pop                    # apply latest and remove
git stash apply stash@{2}        # apply specific, keep in stash
git stash drop stash@{0}         # delete specific stash

# Cherry-pick: apply specific commit(s) to current branch
git cherry-pick abc123
git cherry-pick abc123^..def456  # range (inclusive)

# Bisect: binary search for bug-introducing commit
git bisect start
git bisect bad HEAD              # current commit is bad
git bisect good v1.2.0           # last known good tag
# Git checks out middle commit, you test, then:
git bisect good  # or bad
# Repeat until Git reports the first bad commit
git bisect reset

# Reflog: safety net — all HEAD movements
git reflog
git checkout HEAD@{5}            # restore to 5 HEAD moves ago
git reset --hard HEAD@{3}        # undo to 3 reflog entries ago

Conventional Commits

# Format: type(scope): description
feat(auth): add OAuth2 login with Google
fix(api): handle null user in profile endpoint
docs(readme): update local setup instructions
refactor(db): extract connection pool to module
test(user): add integration tests for registration
perf(feed): use cursor pagination for news feed
ci(github): add Playwright E2E to CI workflow
chore(deps): upgrade express to 5.0

# Breaking change: add ! or BREAKING CHANGE in footer
feat!: remove deprecated v1 API endpoints
# or:
feat(api): new auth system
BREAKING CHANGE: Remove /auth/login, use /auth/token instead

Semantic Versioning

MAJOR.MINOR.PATCH example: 2.1.3
PATCH (2.1.x): backwards-compatible bug fixes
MINOR (2.x.0): backwards-compatible new features
MAJOR (x.0.0): breaking changes
Pre-release: 1.0.0-alpha.1, 2.0.0-rc.3
Build metadata: 1.0.0+20240301

⚡ Key Takeaways

  • git rebase rewrites history (new commit hashes); never rebase commits that others have already pulled
  • git reflog is the safety net for almost any mistake — commits are not permanently deleted for 30 days
  • Conventional Commits (feat:, fix:, chore:) enable automated changelogs and semantic versioning
  • git bisect performs binary search through history to find the commit that introduced a bug
  • Trunk-based development with feature flags reduces merge conflicts and keeps the main branch always releasable
  • PR reviews should focus on design and correctness; style is enforced by linters, not humans
