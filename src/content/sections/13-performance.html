13. Web Performance (Deep) 4 topics

Web performance is a first-class interview topic, especially for senior roles. Core Web Vitals (LCP, CLS, INP) and their measurement, the critical rendering path, bundle optimization, lazy loading, and service workers are all fair game. Always connect performance work to user impact and business outcomes.

13.1 Core Web Vitals
MetricMeasuresGoodNeeds WorkPoor
LCP (Largest Contentful Paint)Loading — when largest element renders< 2.5s2.5s–4s> 4s
INP (Interaction to Next Paint)Responsiveness — worst interaction latency< 200ms200–500ms> 500ms
CLS (Cumulative Layout Shift)Stability — unexpected visual shifts< 0.10.1–0.25> 0.25

Improving LCP

  • Preload the hero image: <link rel="preload" as="image" href="hero.webp" fetchpriority="high">
  • Do NOT lazy-load above-the-fold images
  • Use CDN, eliminate render-blocking resources
  • Serve images in WebP/AVIF format (30-50% smaller)
  • Use fetchpriority="high" on LCP image element

Fixing CLS

  • Always set width and height on images (browser reserves space)
  • Reserve space for ads/embeds with aspect-ratio or min-height
  • Avoid inserting content above existing content after load
  • Use font-display: optional to avoid layout shift from font swap

Fixing INP

  • Break up long tasks (over 50ms) with setTimeout or scheduler.yield()
  • Move non-UI work to Web Workers
  • Use startTransition in React for non-urgent updates
  • Avoid blocking the main thread during input events
13.2 Critical Rendering Path & Script Loading 
// Critical Rendering Path:
// HTML parse -> DOM + CSSOM -> Render Tree -> Layout -> Paint -> Composite

// CSS always render-blocking (blocks paint until parsed)
// Inline critical CSS for above-the-fold content
// <link rel="stylesheet" media="print" onload="this.media='all'"> for non-critical

// Script loading comparison:
// <script src="app.js">          — blocks HTML parsing AND rendering
// <script defer src="app.js">    — downloads in parallel, executes after DOM, in order
// <script async src="app.js">    — downloads in parallel, executes immediately when done (no order guarantee)
// <script type="module">         — deferred by default + strict mode

// Resource hints
<!-- DNS + TCP + TLS pre-setup -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<!-- DNS only -->
<link rel="dns-prefetch" href="https://analytics.example.com">
<!-- High-priority fetch (fonts, hero image) -->
<link rel="preload" as="font" href="font.woff2" crossorigin>
<link rel="preload" as="image" href="hero.webp">
<!-- Idle-time fetch for next navigation -->
<link rel="prefetch" href="/next-page-bundle.js">

13.4 Service Workers & Web Workers 
// Service Worker: intercept network requests, offline caching
self.addEventListener("install", event => {
  event.waitUntil(
    caches.open("v1").then(cache => cache.addAll(["/", "/app.js", "/style.css"]))
  );
});

self.addEventListener("fetch", event => {
  event.respondWith(
    caches.match(event.request).then(cached => {
      const network = fetch(event.request).then(response => {
        caches.open("v1").then(c => c.put(event.request, response.clone()));
        return response;
      });
      return cached || network;  // cache-first
    })
  );
});

// Web Worker: off-main-thread computation
const worker = new Worker("worker.js", { type: "module" });
worker.postMessage({ type: "SORT", data: hugeArray });
worker.onmessage = ({ data }) => displaySortedResults(data);

// worker.js
self.onmessage = ({ data }) => {
  if (data.type === "SORT") {
    const sorted = data.data.sort((a, b) => a - b);
    self.postMessage(sorted);
  }
};

⚡ Key Takeaways

  • LCP measures loading (target <2.5s), CLS measures visual stability (target <0.1), INP measures responsiveness (target <200ms)
  • Preload critical resources (<link rel="preload">); defer non-critical JavaScript (<script defer>)
  • Code splitting and lazy loading reduce initial bundle size; React.lazy and dynamic import() are the standard tools
  • Images should use WebP/AVIF, responsive srcset, and explicit width/height attributes to prevent CLS
  • Service workers enable offline support and instant repeat visits via cache-first strategies
  • Web Workers offload CPU-intensive work (parsing, encryption) off the main thread to prevent jank
