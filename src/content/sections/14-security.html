14. Security (Deep) 5 topics

Security knowledge is expected at mid-level and above. Interviews test HTTPS/TLS, common OWASP vulnerabilities (XSS, CSRF, SQL injection), authentication best practices, CSP headers, and secrets management. For any role touching user data, be ready to discuss data protection by design.

14.1 HTTPS & TLS 
// HSTS: force HTTPS, prevent downgrade attacks
// Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
// Preload: submit domain to browser HSTS preload list

// TLS 1.3 vs 1.2:
// - 1-RTT handshake (vs 2-RTT in 1.2)
// - 0-RTT resumption for repeat connections (replay attack risk — use for safe GETs)
// - Removed weak ciphers: RC4, DES, 3DES, MD5, SHA1
// - Forward secrecy mandatory (ECDHE only)
// - Server certificate encrypted in handshake

// Node.js HTTPS
const https = require("https");
const fs = require("fs");
const server = https.createServer({
  key: fs.readFileSync("private-key.pem"),
  cert: fs.readFileSync("certificate.pem"),
  minVersion: "TLSv1.2"
}, app);
14.2 Content Security Policy (CSP) 
// CSP header: allowlist of trusted content sources
// Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.com 'nonce-RANDOM';

// Key directives:
// default-src: fallback for all content types
// script-src: JavaScript sources
// style-src:  CSS sources
// img-src:    image sources
// connect-src: fetch, XHR, WebSocket destinations
// font-src:   font sources
// frame-ancestors: who can embed this in iframe (replaces X-Frame-Options)
// form-action: where forms can submit
// base-uri:   restricts <base> tag
// upgrade-insecure-requests: auto-upgrade http to https

// Nonce-based scripts (allow specific inline scripts)
// Server generates unique nonce per request
const nonce = crypto.randomBytes(16).toString("base64");
res.setHeader("Content-Security-Policy", `script-src 'nonce-${nonce}'`);
// Template: <script nonce="${nonce}">...</script>

// Report-Only: test without blocking
// Content-Security-Policy-Report-Only: ...; report-uri /csp-report

// Other security headers
// X-Content-Type-Options: nosniff         — prevent MIME sniffing
// Referrer-Policy: strict-origin-when-cross-origin
// Permissions-Policy: camera=(), microphone=(), geolocation=(self)
14.3 Authentication Best Practices 
// Password hashing: adaptive algorithms that are slow by design
import bcrypt from "bcrypt";
const saltRounds = 12; // 2^12 iterations, ~200ms — increase over time
const hash = await bcrypt.hash(password, saltRounds);
const isValid = await bcrypt.compare(password, hash);

// argon2: winner of Password Hashing Competition (preferred over bcrypt)
import argon2 from "argon2";
const hash = await argon2.hash(password, {
  type: argon2.argon2id,  // combines argon2i + argon2d — best for passwords
  memoryCost: 65536,      // 64 MB RAM required
  timeCost: 3,            // iterations
  parallelism: 4          // threads
});
const isValid = await argon2.verify(hash, password);

// NEVER use: MD5, SHA1, SHA256 without salt for passwords (too fast — GPU brute-force)

// JWT security checklist
// 1. Short-lived access tokens: 15 minutes
// 2. Refresh token: long-lived (7d), stored in HttpOnly Secure SameSite=Strict cookie
// 3. Refresh token rotation: issue new refresh token on each use, invalidate old
// 4. Revoke refresh tokens on logout (Redis blacklist or whitelist)
// 5. Always verify algorithm explicitly (prevent alg:none attack)
// 6. Don't store sensitive data in payload (base64-decodable, not encrypted)
// 7. Validate aud, iss, exp claims

const token = jwt.sign({ userId, role }, secret, {
  algorithm: "HS256",     // or RS256 for multi-service
  expiresIn: "15m",
  issuer: "myapp",
  audience: "myapp-api"
});

jwt.verify(token, secret, {
  algorithms: ["HS256"],  // explicit allowlist — prevents alg:none
  issuer: "myapp",
  audience: "myapp-api"
});

Injection Prevention

// SQL Injection
// BAD:
const q = `SELECT * FROM users WHERE email = '${email}'`;
// email = "' OR '1'='1'--" → returns all users

// GOOD: parameterized
await pool.query("SELECT * FROM users WHERE email = $1", [email]);

// Command Injection
// BAD:
exec(`ffmpeg -i ${userFile} output.mp4`); // inject: "; rm -rf /"
// GOOD:
execFile("ffmpeg", ["-i", sanitizedFile, "output.mp4"]);

// Brute force protection
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, max: 10,
  keyGenerator: req => req.body.email || req.ip
});

// Account enumeration: always return same message
// "Invalid email or password" — never "User not found" or "Wrong password"

// Timing attack: always run bcrypt even for unknown users
async function login(email, password) {
  const user = await User.findByEmail(email);
  const dummyHash = "$2b$12$invalidhashpadding000000000000000000000000000000000000";
  const valid = await bcrypt.compare(password, user?.hash || dummyHash);
  if (!user || !valid) throw new UnauthorizedError("Invalid credentials");
  return user;
}

// .env: local development only, in .gitignore
// Production: dedicated secrets manager

// AWS Secrets Manager
import { SecretsManagerClient, GetSecretValueCommand } from "@aws-sdk/client-secrets-manager";

// Rotate secrets regularly

// Never log sensitive data: passwords, tokens, credit card numbers, PII

// Dependency security
npm audit                         // check for known CVEs
npm audit fix                     // auto-fix safe updates
npm audit fix --force             // CAREFUL: may break things
// Dependabot: automated PRs for security updates
// Snyk: deeper scanning, fix PRs, container scanning

// Security headers with helmet.js
import helmet from "helmet";
app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'"],
      scriptSrc: ["'self'"],
      styleSrc: ["'self'", "'unsafe-inline'"],
    }
  },
  hsts: { maxAge: 31536000, includeSubDomains: true, preload: true },
  frameguard: { action: "deny" },
  noSniff: true,
  referrerPolicy: { policy: "strict-origin-when-cross-origin" }
}));

⚡ Key Takeaways

  • XSS: sanitize all user input and use Content Security Policy; innerHTML with untrusted data is dangerous
  • CSRF: use SameSite=Strict cookies and server-side CSRF tokens for all state-changing requests
  • SQL injection: always use parameterized queries — never concatenate user input into SQL strings
  • Store passwords with bcrypt, scrypt, or Argon2 (adaptive, salted hashing) — never MD5 or plain SHA
  • HSTS header forces HTTPS for all future visits; the preload list covers even first-time visitors
  • Rotate secrets regularly; never commit secrets to git — use environment variables and a secrets manager
  • npm audit and Dependabot catch known CVEs in dependencies; treat dependency updates as regular maintenance
